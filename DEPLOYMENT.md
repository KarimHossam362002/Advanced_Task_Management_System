# Deployment Guide

## GitHub Actions + Render

This project is configured to deploy through GitHub Actions after the test workflow passes on `main` or `master`.

### Files Added

- `.github/workflows/deploy-render.yml`
- `render.yaml`

### What This Setup Does

1. GitHub Actions runs the existing `tests` workflow.
2. If tests pass on `main` or `master`, GitHub Actions calls a Render deploy hook.
3. Render rebuilds and deploys the app.

## One-Time Setup

### 1. Push the project to GitHub

Push this repository to GitHub and make sure your default deployment branch is `main` or `master`.

### 2. Create the Render service

In Render:

1. Click **New +**
2. Choose **Blueprint**
3. Connect your GitHub repository
4. Render will detect `render.yaml`
5. Confirm the web service and Postgres database creation

### 3. Get the deploy hook from Render

After the service is created:

1. Open your Render web service
2. Go to **Settings**
3. Find **Deploy Hook**
4. Copy the hook URL

### 4. Add GitHub secret

In GitHub:

1. Open your repository
2. Go to **Settings > Secrets and variables > Actions**
3. Add a new repository secret:

- `RENDER_DEPLOY_HOOK_URL` = your Render deploy hook URL

## How Deployment Works

- Push to `main` or `master`
- GitHub Actions runs tests
- If tests pass, the deploy workflow triggers Render

## Notes

- The app is configured for PostgreSQL in Render, not SQLite
- `APP_KEY` is generated by Render through `render.yaml`
- The free Render web service can sleep when idle
- The free Render Postgres plan is suitable for demos, not long-term production

## Manual Deployment

You can also trigger deployment manually:

1. Open GitHub **Actions**
2. Select **deploy-render**
3. Click **Run workflow**
